Product Requirements Document
EA Overseas — AI-Powered Education Decision Platform
Version 1.0 Status: Draft Owner: Product Head Updated: April 2026

1. Problem Statement

Every year, over 1.3 million Indian students attempt to navigate international and domestic higher education decisions with fragmented, unreliable, and generic guidance. Existing platforms are either purely informational (no personalization), consultant-led (expensive, biased), or job-focused (LinkedIn). No dedicated platform exists that gives students a verifiable, AI-powered, data-driven education identity and admission decision engine.

Core gap: Students waste months researching colleges they are fundamentally unqualified for, and never apply to ones they actually could get into. The decision is made blind.

2. Vision & Core Hypothesis

Build the world's first AI Admission Decision Engine — a platform where a student's complete education profile is matched against real university requirements, producing an honest, actionable compatibility score with a personalized improvement roadmap.

Core value proposition: "Tell me exactly what I need to do — and by when — to get into X university."

3. Target Users

Primary

Indian students, Class 12 to PG, aspiring to study abroad (US, UK, Canada, Germany, Australia)

Secondary

Students targeting Indian institutions (IITs, IIMs, BITS, NIT) needing entrance exam strategy

Tertiary

Test prep coaches, counselors, and university admission teams wanting student lead generation

Security Architecture

  • Auth: JWT (15min access) + HTTP-only cookie refresh tokens (7 days)
  • Data encryption: AES-256 at rest, TLS 1.3 in transit
  • API rate limiting: 100 req/min per user, 10 AI compare calls/day (free tier)

  • Input validation: Zod schemas on all API inputs, parameterized queries (no raw SQL)

AI Match Engine — Technical Spec
Scoring algorithm, GPT integration, and calibration methodology

Architecture: Two-Layer Engine

Layer 1 — Deterministic Scoring Engine: Pure math, weighted formula. Fast (<50ms), consistent, auditable. This is the foundation.
Layer 2 — GPT-4o Explanation Layer: Takes the score breakdown and generates human-readable insights, suggestions, and alternatives. This is the "AI magic" that students see.

Layer 1: Scoring Formula

Final Match Score = 
  (Academic Score   × 0.35) +
  (Test Score       × 0.25) +
  (Budget Fit       × 0.15) +
  (Profile Strength × 0.15) +
  (Extra Factors    × 0.10)

All component scores are normalized 0–100.

Component Scoring Logic

1. Academic Score (weight: 35%)

function academicScore(userGPA, minGPA, avgGPA) {
  if (userGPA >= avgGPA) return 100;
  if (userGPA >= minGPA) {
    return 70 + ((userGPA - minGPA) / (avgGPA - minGPA)) * 30;
  }
  if (userGPA >= minGPA * 0.85) {
    return (userGPA / minGPA) * 70;
  }
  return Math.max(0, (userGPA / minGPA) * 50);
}

2. Test Score (weight: 25%)

function testScore(userScore, requiredScore, avgScore) {
  // Test must exist — if required but missing, return 0
  if (!userScore && requiredScore) return 0;
  if (!requiredScore) return 75; // Not required — neutral

  const ratio = userScore / requiredScore;
  if (ratio >= 1.1) return 100;    // Exceeds by 10%+
  if (ratio >= 1.0) return 90;     // Meets requirement
  if (ratio >= 0.9) return 70;     // Close
  if (ratio >= 0.8) return 45;
  return Math.max(0, ratio * 45);
}
// Multi-test: if university requires IELTS + GRE, score each and average

3. Budget Fit (weight: 15%)

function budgetFit(userBudgetUSD, tuitionUSD, livingUSD) {
  const totalCost = tuitionUSD + livingUSD;
  const ratio = userBudgetUSD / totalCost;

  if (ratio >= 1.0) return 100;   // Can fully afford
  if (ratio >= 0.8) return 80;    // Needs ~20% loan — feasible
  if (ratio >= 0.6) return 55;    // Significant loan needed
  if (ratio >= 0.4) return 30;    // Major funding gap
  return 10;                       // Not viable without scholarship
}

4. Profile Strength (weight: 15%)

function profileStrength(profile) {
  let score = 0;
  
  // Completeness bonus (max 40 pts)
  const completedFields = countCompletedFields(profile);
  score += (completedFields / totalFields) * 40;

  // Qualitative extras (max 60 pts)
  score += Math.min(20, profile.internships * 5);
  score += Math.min(20, profile.researchPapers * 10);
  score += Math.min(10, profile.certifications * 3);
  score += profile.hasWorkExperience ? 10 : 0;

  return Math.min(100, score);
}

5. Extra Factors (weight: 10%)

function extraFactors(profile, courseRequirements) {
  let score = 50; // Base neutral
  
  // Statement of Purpose signals
  if (profile.hasUploadedSOP) score += 15;
  
  // LOR signals
  const lorReady = profile.lorCount >= courseRequirements.lorCount;
  if (lorReady) score += 15;

  // Gap years (mild negative)
  if (profile.gapYears > 2) score -= 10;

  // Country/language match bonus
  if (profile.targetCountries.includes(courseRequirements.country)) score += 10;

  return Math.max(0, Math.min(100, score));
}

Caching Strategy

  • University requirement data: cached in Redis, TTL 6 hours
  • AI comparison results: stored in DB, refreshed only when user profile changes
  • GPT calls: queued via BullMQ, results streamed to frontend via SSE
  • Rate limit: 10 AI compares/day (free), unlimited (paid tier)

Technical Risks Specifically

OpenAI Rate Limits

GPT-4o has rate limits that may throttle during high load. Mitigation: BullMQ queue with retry logic, Redis result cache, fallback to GPT-3.5 for explanation layer only.

Data Quality

University requirement data is messy and frequently changes. Mitigation: structured validation on ingest, quarterly manual review cycle, "Report data error" button on every university page.

Fake Profile Data

Students may inflate scores to game the system. Mitigation: document upload for verification, flag profiles with suspicious patterns, show "unverified" badge.

RDS Cost Blowout

PostgreSQL at scale on RDS is expensive. Mitigation: read replicas for search queries, aggressive indexing, query monitoring from W8, budget alerts on AWS.
