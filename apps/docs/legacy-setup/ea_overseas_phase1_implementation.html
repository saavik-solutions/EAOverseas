Modular architecture
How modules are structured, communicate, and stay decoupled

Principle: Each module owns its own routes, controllers, services, and DB queries. Modules never import from each other's internals — only through a shared lib/ layer. This means any module can be extracted into a separate service later with zero refactoring.

Backend module structure

Every module follows the exact same internal shape. This is non-negotiable — consistent structure means any engineer can navigate any module instantly.

src/modules/{module-name}/
  ├── {module}.routes.ts       ← Fastify route definitions
  ├── {module}.controller.ts   ← Request/response handling only
  ├── {module}.service.ts      ← All business logic lives here
  ├── {module}.repository.ts   ← All DB queries (Prisma calls)
  ├── {module}.schema.ts       ← Zod validation schemas
  ├── {module}.types.ts        ← TypeScript interfaces for this module
  └── {module}.test.ts         ← Unit tests

The 4-layer rule (strict)

Route layer

Only: register path, attach middleware, call controller. Zero business logic. Zero DB access.

Controller layer

Only: parse request, call service, return response. Zero business logic. Zero DB access.

Service layer

All business logic here. Calls repository for data. May call other services via shared lib only.

Repository layer

Only Prisma/DB calls. Returns plain data objects. No business logic. No HTTP concepts.

Shared library (src/lib/)

Cross-cutting concerns that every module can import from. Nothing in here knows about any specific module.

lib/db.ts

Single Prisma client instance. Imported by all repositories.

lib/auth.ts

JWT sign/verify, token utilities. Used by auth middleware.

lib/storage.ts

Cloudflare R2 upload/delete/presign helpers.

lib/email.ts

Resend email client. Send OTP, welcome, reset.

lib/errors.ts

AppError class, error codes, HTTP status mapping.

lib/middleware/

authenticate, requireRole, rateLimiter, requestLogger.

Module — Auth
Registration, login, token management, OTP verification

Scope

auth module

backend

Endpoints this module owns:

  • POST /api/v1/auth/register — create user, send OTP
  • POST /api/v1/auth/verify-otp — verify email OTP
  • POST /api/v1/auth/login — returns access + refresh tokens
  • POST /api/v1/auth/refresh — rotate refresh token
  • POST /api/v1/auth/logout — invalidate refresh token
  • POST /api/v1/auth/forgot-password — send reset link
  • POST /api/v1/auth/reset-password — set new password

Token strategy

Access token:  JWT, 15 min expiry, signed with ACCESS_SECRET
Refresh token: JWT, 7 days expiry, signed with REFRESH_SECRET
               stored in DB (refresh_tokens table) so it can be revoked
               sent as HTTP-only cookie to frontend

Flow:
1. Login → issue both tokens → access in JSON body, refresh in cookie
2. Every request → frontend sends access token in Authorization: Bearer header
3. Access token expired → frontend calls /auth/refresh → gets new pair
4. Logout → delete refresh token from DB → cookie cleared

Auth service (key methods)

// auth.service.ts
class AuthService {
  async register(data: RegisterDto): Promise<User> {
    // 1. Check email not already taken
    // 2. Hash password with bcrypt (rounds: 12)
    // 3. Create user in DB (email_verified: false)
    // 4. Generate 6-digit OTP, store with 10min TTL in Redis
    // 5. Send OTP via email (Resend)
    // 6. Return user (no tokens yet — must verify first)
  }

  async verifyOtp(email: string, otp: string): Promise<TokenPair> {
    // 1. Get OTP from Redis
    // 2. Compare (constant-time)
    // 3. Mark user email_verified: true
    // 4. Delete OTP from Redis
    // 5. Issue access + refresh tokens
  }

  async login(email: string, password: string): Promise<TokenPair> {
    // 1. Find user by email
    // 2. bcrypt.compare(password, hash)
    // 3. Check email_verified — if not, resend OTP
    // 4. Issue tokens
  }

  async refresh(refreshToken: string): Promise<TokenPair> {
    // 1. Verify refresh JWT signature
    // 2. Look up in DB — reject if not found (invalidated)
    // 3. Delete old refresh token row
    // 4. Issue new token pair (rotation)
  }
}

  async function login(email: string, password: string) {
    const res = await authApi.login({ email, password })
    setUser(res.user)
    // access token stored in memory (Zustand)

Profile service (key logic)

// profile.service.ts
class ProfileService {
  async updateAcademics(userId: string, data: AcademicsDto) {
    await this.profileRepo.upsertAcademics(userId, data)
    await this.recomputeStrength(userId)  // always after any update
  }

  async getPresignedUrl(userId: string, fileType: string) {
    const key = `profiles/${userId}/${uuid()}.${ext(fileType)}`
    const url = await storage.getPresignedPut(key, fileType, 3600)
    // Store the key in documents table
    await this.profileRepo.createDocument(userId, key, fileType)
    return { upload_url: url, document_id: key }
  }

  async getPublicProfile(username: string) {
    const user = await this.profileRepo.findByUsername(username)
    if (!user) throw new AppError('USER_NOT_FOUND', 404)
    // Return only public fields — no budget, no documents
    return this.toPublicView(user)
  }
}

Feed repository (pagination pattern)

// feed.repository.ts
async getPosts(params: { page: number; category?: string; userId?: string }) {
  const limit = 20
  const offset = (params.page - 1) * limit

  return prisma.feedPost.findMany({
    where: {
      published: true,
      ...(params.category && { category: params.category }),
    },
    include: {
      _count: { select: { likes: true } },
      // If userId provided, check if this user liked/bookmarked
      likes: params.userId
        ? { where: { user_id: params.userId } }
        : false,
      bookmarks: params.userId
        ? { where: { user_id: params.userId } }
        : false,
    },
    orderBy: [
      { pinned: 'desc' },       // pinned posts always first
      { published_at: 'desc' }, // then latest
    ],
    take: limit,
    skip: offset,
  })
}

Caching strategy for colleges

// colleges.service.ts
async getColleges(filters: CollegeFilters) {
  const cacheKey = `colleges:${JSON.stringify(filters)}`

  // Try Redis first
  const cached = await redis.get(cacheKey)
  if (cached) return JSON.parse(cached)

  // DB query
  const results = await this.collegeRepo.search(filters)

  // Cache for 6 hours (college data changes rarely)
  await redis.set(cacheKey, JSON.stringify(results), 'EX', 21600)

  return results
}

// Cache invalidation: when admin updates any college,
// flush all college:* keys
async invalidateCollegeCache() {
  const keys = await redis.keys('colleges:*')
  if (keys.length) await redis.del(...keys)
}

Admin middleware

// lib/middleware/require-role.ts
export function requireRole(...roles: UserRole[]) {
  return async (request: FastifyRequest, reply: FastifyReply) => {
    // authenticate() must run first
    const user = request.user
    if (!roles.includes(user.role)) {
      throw new AppError('FORBIDDEN', 403)
    }
  }
}

// Usage in admin routes:
fastify.post('/admin/feed',
  { preHandler: [authenticate, requireRole('admin', 'super_admin')] },
  feedController.createPost

API contracts — Phase 1
Request/response shapes for every endpoint. Base URL: /api/v1

Standard response envelope

// All responses follow this shape
{
  "success": true,
  "data": { ... },         // the actual payload
  "meta": {                // pagination (list endpoints only)
    "page": 1,
    "per_page": 20,
    "total": 342,
    "total_pages": 18
  }
}

// Errors
{
  "success": false,
  "error": {
    "code": "INVALID_CREDENTIALS",  // machine-readable
    "message": "Email or password is incorrect",  // human-readable
    "field": "email"  // optional — for validation errors
  }
}

Auth endpoints

POST /auth/register
Body:  { full_name, email, password }
200:   { data: { user: { id, email, full_name, username } } }
409:   EMAIL_ALREADY_EXISTS

POST /auth/verify-otp
Body:  { email, otp }
200:   { data: { access_token, user } } + Set-Cookie: refresh_token
400:   INVALID_OTP / OTP_EXPIRED

POST /auth/login
Body:  { email, password }
200:   { data: { access_token, user } } + Set-Cookie: refresh_token
401:   INVALID_CREDENTIALS
403:   EMAIL_NOT_VERIFIED

POST /auth/refresh
Cookie: refresh_token (HTTP-only)
200:   { data: { access_token } } + new Set-Cookie: refresh_token
401:   INVALID_REFRESH_TOKEN
